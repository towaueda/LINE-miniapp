  1. 3人揃わなかった場合                                                                                                                                
                                                                                                                                                        
  現実的な対処法                                                                                                                                        
                                                                                                                                                        
  ① 待機期限 + 自動延長                                           
  - リクエストに有効期限（例：7日間）を設定
  - 期限が近づいたら「まだマッチング待ちです」とLINE通知
  - 期限切れ時は自動キャンセル + 「再応募しますか？」を送信

  ② 2人マッチングへのフォールバック
  - 待機〇日後、2人でもマッチングするかユーザーに選択肢を提示
  - ただしコンセプト（3人での出会い）が変わるのでユーザー同意が前提

  ③ 管理者による手動補完
  - 待機者が2人いる状況で、運営が「サクラ」ではなくスタッフが同席して3人にする
  - 小規模運営段階では現実的

  推奨：① + 期限切れ通知が最もシンプル

  現在すでに expire_old_match_requests RPCがあるので、これに通知を追加するだけで実装可能です。

  2. 同じ日時に5人集まった場合

  振り分けの考え方

  現在のシステム（3人固定）だと：
  - 5人 → 3人グループ + 2人は待機という中途半端な状態になる

  選択肢

  ┌─────────────────────────────────────────────────┬──────────────────────────────────┬──────────────────────────────┐
  │                    パターン                     │             メリット             │          デメリット          │
  ├─────────────────────────────────────────────────┼──────────────────────────────────┼──────────────────────────────┤
  │ 先着3人でグループ確定                           │ シンプル、実装済みロジックに近い │ 残り2人が不満                │
  ├─────────────────────────────────────────────────┼──────────────────────────────────┼──────────────────────────────┤
  │ 3+2に分けて、2人は次回優先枠                    │ 公平感ある                       │ 複雑、優先枠の管理が必要     │
  ├─────────────────────────────────────────────────┼──────────────────────────────────┼──────────────────────────────┤
  │ 5人→3人+2人で2グループ（2人グループを別途作る） │ 全員参加できる                   │ コンセプト変更が必要         │
  ├─────────────────────────────────────────────────┼──────────────────────────────────┼──────────────────────────────┤
  │ ランダムで3人を選ぶ                             │ 公平                             │ 選ばれなかった人の体験が悪い │
  └─────────────────────────────────────────────────┴──────────────────────────────────┴──────────────────────────────┘

  推奨：「先着3人 + 残り2人は次回優先」

  - マッチング順：リクエスト作成時刻が早い3人を優先（現行のatomic RPCの動作）
  - 残り2人には通知「今回は惜しくも外れましたが、次回優先マッチングの権利を付与しました」
  - DBに priority フラグを追加してマッチング時に優先

  これが公平でユーザー体験も悪くないバランスだと思います。
