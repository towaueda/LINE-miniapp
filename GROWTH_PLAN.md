# Triangle 改善・ユーザー獲得プラン

作成日: 2026-03-31
最終更新: 2026-04-02（修正提案反映）

---

## 現状の課題

- 本番公開翌日時点でユーザー3人
- プロフィール未入力のまま止まっている
- 原因: 登録フローのActivation Energy（着手コスト）が高すぎる + 信頼の欠如

---

## 信頼構築（Trust Building）

### 優先度: 🔴 高（今日中）— フリクション削減より先に取り組む

フリクションを下げても「怪しい」と思われたら登録しない。信頼が土台。

#### T-1. 開発者の顔出し・素性の公開
- **問題**: 作者が誰か分からない → 「情報を抜かれるかも」という不安が先行する
- **改善**: ホーム画面の下部または利用規約ページに開発者プロフィールを追加
  ```
  👤 開発者: ○○ ○○（大阪在住・会社員）
  「ランチ難民を解消したくて作りました」
  X: @xxxx / note: xxxx
  ```
- **根拠**: 顔と名前が見える人間に情報を渡すことへの抵抗は大幅に下がる（Liking の原理）
- **注意**: SNSリンクがあると「逃げ場がある人」として信頼される

#### T-2. 取得情報の3行宣言（登録フローの冒頭）
- **問題**: 「どんな情報を取られるか分からない」不安が登録を止める
- **改善**: 利用規約より先に、登録画面の上部に表示
  ```
  このアプリが取得するのは:
  ✅ LINEの表示名とプロフィール画像
  ✅ あなたが入力した自己紹介・職種・エリア
  ❌ LINEトーク・電話番号・友だちリストは取得しません
  ```
- **根拠**: 不安の正体を言語化して消す（Uncertainty Reduction Theory）

## UX改善（フリクション削減）

### 優先度: 🔴 高（今日中）

#### 1. 利用規約モーダルをチェックボックスに変更
- **現状**: モーダルを最後までスクロールしないと同意ボタンが押せない
- **問題**: 最大の離脱ポイント。スクロール強制はストレスが高く、早期段階では致命的
- **改善**: チェックボックス1つに置き換える
  ```
  □ 利用規約・プライバシーポリシーに同意します（リンク）
  ```
- **根拠**: BJ Fogg モデル — Ability（やりやすさ）の向上

#### 2. プロフィール進捗バーの追加
- **現状**: 何項目あるか・あと何項目かが見えない
- **改善**: ページ上部に進捗を表示
  ```
  プロフィール登録  ████████░░  5 / 6 完了
  ```
- **根拠**: Goal-Gradient Effect — ゴールが近いと感じるほど完走率が上がる

#### 3. 会社名/屋号を任意項目にする
- **現状**: 会社名が必須（`isValid` に含まれている）
- **問題**: 初対面のサービスに会社名を入力することへの心理的抵抗が強い
- **改善**: 「任意」に変更し、`isValid` のチェック対象から外す
- **根拠**: Paradox of Choice — 必須項目を減らすと完了率が上がる

#### 4. 自己紹介(bio)にサンプル文を表示
- **現状**: placeholder が「趣味や好きなものを書いてみましょう」のみ
- **問題**: 何を書けばいいか迷って止まる（認知的負荷）
- **改善**: 入力欄の下にタップして挿入できるサンプルを表示
  ```
  例: 「IT系で働く29歳。週末はキャンプ派です」
      「メーカー営業3年目。ランチ難民中...」
  ```
- **根拠**: Activation Energy の削減 — 「書く」から「選ぶ」に変換

---

### 優先度: 🟠 中（今週中）

#### 5. プロフィール完了後に「友達招待」を促す
- **現状**: プロフィール保存 → マッチングページへ遷移するだけ
- **改善**: 完了画面に「招待コードを友達にシェアする」ボタンを追加
- **根拠**: Reciprocity（返報性）+ IKEA Effect — 自分で作ったプロフィールへの愛着がシェア動機になる

#### 6. ホーム画面に社会的証明を追加
- **現状**: 機能説明カード3枚のみ
- **改善**: 「現在〇人が参加中」「今週〇件のランチが成立」などを表示
- **根拠**: Social Proof / Bandwagon Effect — 人が使っているサービスは安心感がある
- **注意**: ユーザーが少ないうちは「参加受付中」など数字を出さない表現でもOK

#### 7. プロフィール未完了ユーザーへの自動リマインド
- **現状**: リマインド機能なし
- **改善**: 登録から24時間以内にプロフィール未完了ならLINE通知を送るcron jobを追加
  ```
  【Triangle】プロフィールがまだ未完了です。
  2分で完成できます👇
  ```
- **根拠**: Zeigarnik Effect — 未完了タスクは完了したくなる心理的引力がある

## 心理的障壁（追加分析）

登録・プロフィール入力を嫌がる心理は「面倒くさい」だけではない。

### 障壁1: 公開範囲への不安（Privacy Calculus）
- **心理**: 「入力した情報がどこまで他人に見えるのか分からない」
- **改善**: 各入力項目の横に公開範囲を明示する
  ```
  職種 [IT・Web系 ▼]  👥 マッチング相手に公開
  年齢  [29]           👤 自分のみ（マッチングの内部計算に使用）
  ```
- **根拠**: 何が起きるかを可視化するだけで入力完了率が上がる

### 障壁2: 後戻りできない感（Commitment Fear）
- **心理**: 「登録したら抜けられなくなるのでは」「しつこい通知が来るのでは」
- **改善**: 登録画面のどこかに一言添える
  ```
  いつでも退会・情報削除できます
  ```
- **根拠**: 出口が見えると入口のハードルが下がる（Reactance Theory）

### 障壁3: 自己開示への恥ずかしさ（Self-Presentation Anxiety）
- **心理**: 「自己紹介文を書くのが恥ずかしい・うまく書けない」→ 特にビジネス系ユーザー
- **改善**: 「あとで編集できます」をプロフィール画面に表示 + bioを任意化を検討
  ```
  ✏️ プロフィールはあとからいつでも変更できます
  ```
- **⚠️ 意思決定が必要**: bioの任意化について「検討」のままになっている。現状コード（`profile/page.tsx:43`）では `bio.trim()` が `isValid` に含まれており必須。任意化するか否かを明示的に決定し、今週タスクに反映すること。サンプル文追加だけでは障壁が残る可能性がある。
- **根拠**: 完璧主義による離脱を防ぐ。「仮入力でいい」と思わせる

### 障壁4: サービス価値が入力前に見えない（Value Before Effort）
- **心理**: 「入力してみたけど誰ともマッチングしなかったら時間の無駄」
- **改善**: プロフィール入力フローの最初に、入力後に得られる体験を見せる
  ```
  プロフィールを登録すると…
  → あなたのエリア・職種に近い人とランチがマッチングされます
  → 毎週○曜日に候補が届きます
  ```
- **根拠**: 投資対効果を先に示す（Value Proposition Clarity）

### 障壁5: 「自分が使うサービスじゃない」という思い込み（Identity Misfit）
- **心理**: 「ランチを知らない人と食べるのは自分には向いていない」と思っている
- **改善**: ホーム画面のコピーに具体的なペルソナを入れる
  ```
  ❌ 「新しいつながりを作ろう」（抽象的）
  ✅ 「在宅勤務でランチが孤独な人・転職したばかりで知り合いが少ない人に」
  ```
- **根拠**: 自分ゴト化できると行動に移しやすい（Self-Referencing Effect）

## アクティブユーザーを増やす施策

### OMTM（唯一追うべき指標）
> **「月間ランチ完了数」** — 登録数・DAUはすべて虚栄指標。ランチが完了して初めて価値が出る。

### 構造的問題（最優先で直す）

> ⚠️ **注意**: S-1〜S-3は「最優先」と位置づけているが、元のチェックリストに未記載だった。今週タスクに追加済み（下部参照）。

#### S-1. 木曜限定がマッチング確率を壊している
- **問題**: 月4回しかチャンスがない。ユーザーが3人揃う確率が低すぎて、NoMatchViewで終わる確率が高い
- **改善**: 木・金の2日に拡大。それだけで月あたりのマッチング機会が2倍になる
- **実装箇所**: `matching/page.tsx` の `getNextThursdays()` を `getNextThursdaysAndFridays()` に変更（`d.getDay() === 4 || d.getDay() === 5`）+ `vercel.json` のcronスケジュールも要変更
- **根拠**: Leaky Bucket — マッチング失敗 = 即離脱。入口を広げる前にマッチング成功率を上げる

#### S-2. マッチング待ち中にアプリを閉じると詰む
- **問題**: `WaitingView`は5秒ごとにポーリングしているが、アプリを閉じると機能しない。マッチングが成立してもユーザーが気づかない可能性がある
- **現状コード**: `isWaiting`フラグ + `setInterval(checkStatus, 5000)` — フォアグラウンド前提
- **⚠️ 緊急対応**: WaitingView の「3人揃ったらお知らせします！」という文言は **Push通知未実装の現状では事実と異なる**。S-2実装前は「アプリを開いて確認してください」等の文言に変更すること（E-1と合わせて対応）
- **改善**: マッチング成立時にLINE Bot Push通知を送る（`/api/cron/matching`のタイミングで通知）
  ```
  【Triangle】マッチング成立！
  ○○さん・△△さんと木曜ランチが決まりました 🎉
  アプリで詳細を確認してください👇
  ```
- **実装箇所**: `src/app/api/cron/matching/route.ts` → `offerTwoPersonMatches()` 完了後にLINE Bot pushを呼ぶ処理を追加
- **根拠**: アプリを閉じた後も再来訪させる唯一の手段

#### S-3. NoMatchViewが完全な袋小路
- **問題**: 「マッチングなし😔 別の日程で探す」→ 何も残らない。このまま離脱するユーザーが多い
- **改善**: 「マッチングなし」のときに「同じエリアで待っている人が出たらLINEでお知らせします」オプションを追加。メール/LINEで再通知を約束する
  ```
  📩 梅田エリアに参加者が増えたらLINEでお知らせする
  [はい、通知を受け取る]
  ```
- **実装コスト**: DBに `notify_on_match_available` フラグ＋エリアカラムを追加し、cron側で通知処理が必要。工数中〜高。バックログ扱い。
- **根拠**: Zeigarnik Effect — 未完了のマッチングを未解決にしておくと再来訪率が上がる

### エンゲージメント・リテンション

#### E-1. WaitingViewの文言修正＋「閉じていいよ」を伝える ⚠️ 今日対応
- **現状**: スピナーが回り続けてユーザーを画面に縛る。かつ「3人揃ったらお知らせします！」という**Push通知未実装の状態では事実と異なる文言**が表示されている
- **改善フェーズ1（今日・S-2実装前）**: 文言を実態に合わせる
  ```
  マッチング待ち中です。
  成立したらこのページに戻って確認してください。
  ```
- **改善フェーズ2（S-2実装後）**: Push通知が使えるようになったら差し替える
  ```
  マッチングが成立したらLINEでお知らせします。
  このページは閉じても大丈夫です。
  ```
- **実装箇所**: `matching/page.tsx` の `WaitingView` コンポーネント（約350行目）
- **根拠**: 不安を取り除くだけで満足度が上がり、次回の申請意欲が維持される

#### E-2. レビュー完了後の招待コード導線を強化する
- **現状**: レビュー後に招待コードが発行されるが「高評価ユーザーとして〜」という文言で終わる
- **問題**: なぜ友達に渡すべきかの動機が弱い（コードの価値が見えない）
- **改善**: 招待コードの表示を「友達が使うと○○になる」に変える
  ```
  このコードを渡した友達がランチを完了すると、
  あなたの次回マッチングが優先されます 🔺
  ```
- **⚠️ 実装コスト注意**: 「優先マッチング」を実現するにはマッチングロジックに優先度フラグの追加が必要で工数が大きい。代替として「次回ランチの費用を一部負担」「コーヒー券プレゼント」など物理的インセンティブも検討すること
- **根拠**: Two-sided incentive — 渡す側にもベネフィットがあると拡散率が上がる

#### E-3. プロフィールに「ランチ実績バッジ」を表示する
- **現状**: プロフィールは静的。成長感がない
- **改善**: 完了ランチ数・評価を簡単に表示
  ```
  🍱 ランチ実績: 3回
  ⭐ 平均評価: 4.2
  ```
- **根拠**: Progress & Achievement — 実績の積み上がりがサービスへの投資感を生み、継続率を上げる

#### E-4. ホーム画面に「待機中の人数」をリアルタイム表示
- **現状**: FeatureCard 3枚のみ。サービスが生きているか分からない
- **改善**: ログイン後のマッチングページ上部に表示
  ```
  📍 梅田エリアで今 2人 が木曜ランチを待っています
  ```
- **⚠️ 注意（S-6と同様）**: エリアあたりの待機人数が少ない現段階（〜5人未満）では「2人しかいない」という印象を与えて逆効果になりうる。**エリアあたり5人以上になるまでは数字ではなく「参加受付中」等の表現にとどめる**こと
- **根拠**: FOMO + Social Proof — 「今申請しないと機会を逃す」と感じさせる

#### E-5. 木曜当日のリマインド通知
- **問題**: ランチ当日に「今日だよ」と気づかせる仕組みがない
- **改善**: マッチング成立後、当日朝9時にLINE通知
  ```
  【Triangle】今日12時にランチです！
  📍 梅田 ○○レストラン
  △△さん・□□さんと会いましょう 🍽️
  ```
- **根拠**: Reminder × Anticipation — 当日キャンセルを減らし、体験品質を上げる

## ユーザー獲得

### コールドスタート問題
3人マッチングは**一定数のユーザーがいないと価値が出ない**。
まず梅田・本町・難波エリアで**2026年4月末までに**20〜30人を集めることを最初のゴールにする。

### 獲得チャネル（優先順）

| 優先度 | チャネル | 具体的アクション |
|--------|---------|----------------|
| 🔴 高 | 知人への直接配布 | 招待コードを10〜20枚、ビジネス系の知人に直接渡す |
| 🔴 高 | LINE Botブロードキャスト | 現在の3ユーザーにプロフィール完了を促すメッセージを送る |
| 🟠 中 | コミュニティへの告知 | 大阪のビジネス系Slack/Discord/Facebookグループに投稿 |
| 🟠 中 | note/Xでの発信 | 「3人ランチアプリを作った」開発ストーリーを書く |
| 🟡 低 | オフライン | 梅田・本町エリアのコワーキングスペースに声がけ |

### バイラルループの設計
今の招待コードは「入場チケット」として機能しているが、**友達を招待した動機が薄い**。

改善の方向性:
- プロフィール完了時に自分の招待コードが発行される → 「友達と一緒に使いたい」動機が生まれる
- 招待した友達がランチを完了したら通知が届く → 招待の達成感が生まれる

## 効果測定（毎週確認する数値）

施策を打っても計測できないと次の改善につながらない。Supabaseで毎週月曜に確認する。

| 指標 | 確認方法 | 目標（4月末） |
|------|---------|-------------|
| 総ユーザー数 | `users` テーブルのレコード数 | 20〜30人 |
| プロフィール完了率 | `nickname` + `area` + `industry` + `bio` がすべて入っている割合 | 80%以上 |
| マッチング申請数/週 | `matching_requests` テーブル | 週5件以上 |
| **月間ランチ完了数（OMTM）** | `matches` テーブルの `status = 'completed'` | 月4件以上 |
| NoMatch率 | `status = 'no_match'` / 全申請数 | 50%以下 |
